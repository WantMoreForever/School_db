# EduManage 教学管理系统答辩说明

本文档用于 `EduManage` 教学管理系统的课程设计答辩、课堂演示与项目介绍，重点说明项目定位、系统结构、数据库设计亮点、演示路线与答辩口径。

## 1. 项目定位

`EduManage` 是一个面向高校教学场景的教学管理系统，围绕管理员、教师、学生三个角色构建完整业务闭环：

- 管理员维护人员、课程、院系、专业、教室、排课、公告和系统日志；
- 教师管理任教课程、考试、成绩、考勤、课程公告和工作量；
- 学生查看门户、选课、课表、成绩、考试安排、空闲教室和通知公告。

项目支持 PHP + MySQL 的 Web 环境、本地集成环境、Apache 或 Nginx，适合作为课程项目进行展示和讲解。

## 2. 系统结构

```text
浏览器
  |
  |  HTML / CSS / JavaScript
  v
前端页面层
  |-- login/
  |-- admin/
  |-- teacher/
  |-- student/
  |
  |  PHP Session + HTTP API
  v
后端接口层
  |-- admin/api/
  |-- teacher/api/
  |-- student/api/
  |-- components/
  |
  |  PDO
  v
MySQL 数据库
  |-- 表
  |-- 视图
  |-- 函数
  |-- 存储过程
  |-- 触发器
  |-- 索引
```

答辩时可以概括为：

- 应用层负责页面交互、权限控制与接口编排；
- 数据库层负责核心结构、规则校验、业务计算与过程封装；
- 三端围绕统一账号体系协同运行。

## 3. 数据库设计亮点

### 3.1 初始化主脚本明确

`school_db_backup.sql` 是数据库初始化主脚本，也是当前数据库结构与对象的权威来源。导入该脚本后，系统可以一次性恢复：

- 21 张数据表；
- 5 个视图；
- 7 个函数；
- 60 个存储过程；
- 16 个触发器；
- 当前使用的关键索引。

之后再导入 `tests/sql/demo_seed.sql` 即可得到答辩演示账号与演示数据。

### 3.2 核心教学业务采用存储过程封装

这是本项目答辩时最值得强调的设计点。

系统将以下核心教学业务封装在存储过程中：

- 学生可选课程、已选课程、课表与门户摘要；
- 教师任教班级、教师课表和导师关系；
- 考试发布、考试事件管理、成绩录入、成绩统计与字母成绩转换；
- 考勤记录、考勤报表与教师工作量统计；
- 公告可见范围与课程公告读取；
- 空闲教室与冲突查询。

这说明数据库层不仅保存数据，也承担了业务规则与过程控制。

### 3.3 视图、函数、触发器、索引配套完整

除存储过程外，数据库还通过以下对象增强系统表达力：

- 视图：为课表、班级概览、教师聚合、成绩展示提供稳定数据视角；
- 函数：实现 GPA、成绩映射、冲突检测、出勤率等计算；
- 触发器：校验编码、联系方式、成绩范围、公告投放和选课时间；
- 索引：围绕选课、排课、成绩、公告和日志等高频链路优化。

## 4. 核心 ER 关系

```text
user
  |-- admin
  |-- teacher -- teaching -- section -- course
  |      |                     |
  |      |                     |-- schedule -- classroom
  |      |                     |-- exam
  |      |                     |-- attendance
  |      |
  |      |-- advisor -- student
  |
  |-- student -- takes -- section

department -- major -- student
department -- teacher

announcement -- announcement_target
system_log
```

讲解时可以突出：

- `user` 是统一账号主表；
- `section` 是“教学业务连接点”，把课程、教师、学生、排课、成绩、公告联系起来；
- `takes`、`exam`、`attendance` 共同构成学生学习过程数据；
- `announcement_target` 支持公告精准投放。

## 5. 推荐演示路线

### 5.1 管理员端

1. 登录管理员账号；
2. 展示后台首页统计；
3. 进入学生管理或教师管理，说明人员维护；
4. 进入课程、教室、院系、专业管理，展示基础教学数据治理；
5. 进入排课管理，讲解课程、教师、教室、周次与节次的关系；
6. 发布一条公告，说明公告投放目标；
7. 打开系统日志，展示操作记录。

### 5.2 教师端

1. 登录教师账号；
2. 展示教师首页仪表盘；
3. 查看任教班级与学生名单；
4. 演示发布考试、录入成绩、自动分配字母成绩；
5. 演示按排课与周次记录考勤；
6. 查看课程公告与工作量统计。

### 5.3 学生端

1. 登录学生账号；
2. 展示门户首页中的 GPA、学分、选课数和最近成绩；
3. 进入选课系统，展示可选课程、已选课程与容量信息；
4. 查看课表与考试安排；
5. 进入成绩查询；
6. 查看空闲教室；
7. 打开通知公告，验证公告精准投放效果。

## 6. 答辩讲解重点

建议围绕下面五点展开：

1. 项目不是单页面展示，而是管理员、教师、学生三端完整闭环；
2. `school_db_backup.sql` 是数据库初始化主脚本，部署和演示主线清晰；
3. 存储过程封装了核心教学业务，体现数据库课程设计深度；
4. 视图、函数、触发器、索引配套完整，展示数据库综合设计能力；
5. 文档、演示数据和 smoke 测试齐备，交付性较强。

## 7. 常见提问与回答口径

### 为什么把业务写进存储过程？

答：因为选课、课表、考试、成绩、考勤和工作量等业务都涉及多表联动与统一规则。把这些逻辑放进存储过程，能够让规则集中、查询复用、演示讲解更清晰。

### `school_db_backup.sql` 的作用是什么？

答：它是数据库初始化主脚本，包含当前项目运行所需的表、视图、函数、存储过程、触发器和索引，是恢复数据库环境的主入口。

### 项目的数据库亮点在哪里？

答：亮点在于数据库对象比较完整，不仅有表结构，还有视图、函数、存储过程、触发器和索引，并且这些对象真实参与了教学业务流程。

### 为什么这个项目适合课程答辩？

答：因为它既有完整角色业务闭环，也有较完整的数据库设计表达，还具备部署文档、演示数据和测试脚本，容易展示“设计思路 + 实现结果 + 可运行性”。
