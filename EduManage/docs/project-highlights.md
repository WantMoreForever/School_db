# EduManage 项目亮点

## 1. 文档定位

本文档用于概括当前 `EduManage` 的项目亮点与交付价值，适合教师查阅、课程答辩和项目汇报。所有内容均以当前代码、目录结构与 `school_db_backup.sql` 为准，不描述未实现功能。

## 2. 项目亮点

### 2.1 三角色教学闭环完整

项目已经形成管理员、教师、学生三端闭环：

- 管理员负责基础数据、人员、排课、公告和日志治理；
- 教师负责任教课程、考试、成绩、考勤、公告和工作量；
- 学生负责门户、选课、课表、成绩、考试、空闲教室和公告。

真实入口清晰：

- `/admin/index.php`
- `/teacher/index.php`
- `/student/spa.html`

### 2.2 当前目录结构清晰，适合教学展示

项目目录围绕业务角色组织：

- `admin/`、`teacher/`、`student/` 分别对应三端；
- `login/` 负责统一登录入口；
- `components/` 沉淀公共数据库、认证、日志与成绩辅助逻辑；
- `config/` 集中管理数据库、路径、权限、上传和前端配置；
- `docs/` 已形成部署、数据库、亮点、答辩等文档体系。

这种结构便于教师快速理解项目边界，也便于后续维护。

### 2.3 `school_db_backup.sql` 不是普通备份，而是初始化主脚本

项目数据库初始化主线非常明确：

1. 导入 `school_db_backup.sql`
2. 导入 `tests/sql/demo_seed.sql`

其中 `school_db_backup.sql` 包含当前系统的：

- 21 张数据表；
- 5 个视图；
- 7 个函数；
- 60 个存储过程；
- 16 个触发器；
- 当前使用的关键索引定义。

这意味着项目数据库可以通过一份主脚本快速恢复到可演示状态，适合课程答辩和环境重建。

### 2.4 核心教学业务采用存储过程封装

这是本项目最值得展示的数据库设计亮点之一。

系统将以下核心教学业务封装到存储过程中：

- 选课资格与可选课程读取；
- 学生课表与门户摘要；
- 教师课表、任教班级与导师关系；
- 考试发布、成绩录入、成绩统计与字母成绩生成；
- 考勤记录与考勤报表；
- 教师工作量统计；
- 公告可见范围过滤与课程公告读取；
- 空闲教室与冲突查询。

这样的设计体现出数据库层不只是“存数据”，而是在承担教学业务规则和复用查询能力。

### 2.5 视图、函数、触发器、索引协同设计完整

除存储过程外，项目数据库还具备完整的配套对象：

- 视图：为课表、班级概览、教师聚合、成绩展示提供稳定视角；
- 函数：用于 GPA、成绩映射、冲突检测、出勤率等计算；
- 触发器：用于编码、联系方式、成绩范围、公告投放、选课时间等规则校验；
- 索引：围绕选课、排课、成绩、公告和日志等高频查询链路优化。

这使项目具备比较完整的数据库课程设计深度。

### 2.6 不是简单 CRUD，而是面向教学场景的业务链路

项目中的关键模块已经体现出真实教学业务特征：

- 排课需要同时考虑课程、教师、教室、周次和时间段；
- 选课要结合时间冲突、容量、开放时间和专业限制；
- 成绩管理不仅包含分数录入，还包含考试事件、字母成绩和统计分析；
- 公告不是单表发布，而是支持全体、教师、学生、专业和班级的精准投放。

### 2.7 部署与交付口径清晰

项目支持 PHP + MySQL 的 Web 环境、本地集成环境、Apache 或 Nginx。部署时只需围绕以下主线即可完成：

- 准备 PHP + MySQL 运行环境；
- 导入 `school_db_backup.sql` 与 `tests/sql/demo_seed.sql`；
- 配置 `config/database.php` 与 `config/app.php`；
- 检查三端入口和演示账号；
- 运行 smoke 测试做演示前核查。

这对课程项目交付非常友好。

### 2.8 文档体系适合答辩展示

当前仓库已经形成分层文档：

- `README.md`：项目总览；
- `PROJECT_FUNCTION_DATABASE_DOC.md`：功能与数据库总体说明；
- `docs/deployment-guide.md`：部署与验收；
- `docs/database-schema.md`：数据库结构说明；
- `docs/defense-guide.md`：答辩讲解与演示路线。

这让教师可以根据需要快速进入对应层级，而不是只看一份杂糅的大文档。
